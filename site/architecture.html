tono · repo guide · v2.0.0

One engine, several faces

tono is a deterministic synthesis-graph engine: you describe a sound as a JSON document, the engine renders it to audio and hands back a spectrogram, a waveform, and numeric stats — so sound is authored by inspection, in a write → render → look → refine loop. Everything else in the repo is a different way of driving that one engine.

The invariant that shapes everything
render(graph, seed, sample_rate) → byte-identical audio, forever

The same document renders the same bytes on every run — and the real-time streaming path matches the offline bounce bit-for-bit. A golden corpus (crates/tono-core/tests/golden.rs) pins the hashes in CI. Kernel improvements that would change output ship behind a document engine revision (currently 4), so old sounds never silently change. If a change you make shifts a golden hash, that is the suite telling you that you changed synthesis math, not organisation.

the map

Cargo workspace. The root crate is the CLI; the pure engine and the native faces live under crates/. The three cpal-based crates are off the default build — make verify never compiles them; make verify-native is their gate.

The one seam to remember: runtime::AudioSource — “fill this interleaved-stereo buffer.” Every live thing (Engine, Mixer, Instrument, AdaptiveMusic, the streaming renderer) implements it, and every output adapter (the CLI’s bounce excepted) only ever talks to that trait. It is why the engine never depends on a sound card.

how a sound flows

you write            the engine                       you look at
─────────────        ──────────────────────────       ─────────────────
blip.json   ──►  dsl: parse + validate  ──►  render  ──►  blip.wav
{ "root": {          (loud errors, NaN-proof)   │         blip.png       ◄─ spectrogram
  "type":"mul",                                 │         blip_wave.png
  "inputs":[…] }}                               └──►  analysis ──►  blip.stats.json
        ▲                                                      │
        └──────────────  refine the JSON, re-render  ◄─────────┘

That loop is the product. The CLI (tono render) is its most direct form; the desktop studio runs the same loop per grid edit; tono-py runs it from Python. The live paths (streaming, runtime) exist so what you audition in real time is bit-for-bit what the offline bounce will ship.

start here

  1. Build and prove the invariant to yourself

    One command runs exactly what CI runs — fmt, clippy, all 250+ tests including the golden corpus and the offline/streaming byte-identity fuzz.

    make verify        # the default-build gate
make hooks         # install pre-commit / pre-push gates once per clone

  2. Render your first document

    Write a doc, render it, open the PNGs. This is the authoring loop every face wraps.

    cat > blip.json <<'EOF'
{ "name": "blip", "duration": 0.3, "engine": 4,
  "root": { "type": "mul", "inputs": [
    { "type": "sine", "freq": 880 },
    { "type": "env", "a": 0.002, "d": 0.08, "s": 0.0, "r": 0.05 } ] } }
EOF
cargo run -- render blip.json -o out/
open out/blip.png    # the spectrogram; out/blip.stats.json has the numbers

  3. Learn the node vocabulary

    docs/cookbook.md is the DSL reference — every node type, the engine revisions, recipes for SFX, loops, and full songs. Skim it with crates/tono-core/src/dsl/node.rs open; the doc comments are the source of truth.

  4. Hear it live from code

    The tono-play examples are the guided tour of the higher layers — sounds, instruments, voice management, buses, songs, adaptive game music.

    make play                            # the playground walkthrough
make play EXAMPLE=band               # catalog + fluent song builder
make play EXAMPLE=adaptive           # intensity-driven stems + stinger
ls crates/tono-play/examples/        # all ten

  5. Read the core in dependency order

    Each layer only uses the ones before it: dsl → render → streaming → runtime → instrument/adaptive → song/catalog. Start at dsl/mod.rs (SoundDoc) and render/mod.rs (render_product) — everything else is a consumer of those two.

  6. Make a change safely

    Adding a node type touches four places: dsl/node.rs (the variant), dsl/validate.rs (its checks), render/ (offline kernel), streaming/ (real-time mirror — or let it fall back to the Player). Changing how an existing node sounds is different: that must go behind a new engine revision so old documents keep their bytes. The golden tests arbitrate.

    make verify          # default crates (CI: ubuntu + macos)
make verify-native   # tono-desktop / tono-play / tono-py + all examples
